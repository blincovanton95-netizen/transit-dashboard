## 🎯 Промпт 1: Транспортная карта с плавной анимацией

### Исходный промпт
```text
Роль: Senior Vue 3 Developer, эксперт по Leaflet и real-time визуализации.
Задача: Создать компонент TransportMap.vue для отображения 100+ движущихся маркеров транспорта.

КРИТИЧНЫЕ требования по производительности:
1. Маркеры должны обновляться каждые 5 секунд БЕЗ пересоздания DOM-элементов
2. Использовать Map<id, L.Marker> для O(1) доступа к маркерам
3. Двигать существующие маркеры через setLatLng(), а не пересоздавать
4. Интерполировать координаты между обновлениями через requestAnimationFrame для плавности
5. Применять v-memo или эквивалент для минимизации реактивных обновлений
6. Использовать L.layerGroup() для группировки маркеров

Контекст:
- Данные приходят через SSE (Server-Sent Events)
- Каждое ТС имеет: id, type, coordinates {lat, lon, bearing, speed}, status, delay
- При клике на маркер — popup с деталями
- Поддержка фильтров по типу ТС, маршруту, статусу
- Цветовая индикация: зелёный (движется), жёлтый (на остановке), красный (опоздание)

Требования к UX:
- Hover на маркер → увеличение (scale 1.2)
- Задержка > 5 мин → красный бейдж "!" на иконке
- Подсветка маркера при вызове из чат-бота (через $onAction)
- Плавное центрирование карты на выбранном ТС

Формат: ТОЛЬКО код TransportMap.vue с комментариями по оптимизациям.

Почему был сложным

- Конфликт между реактивностью Vue и производительностью Leaflet
Vue хочет реактивно обновлять DOM, но Leaflet управляет своим DOM через canvas/SVG. Нужно найти баланс.
- Термин "интерполяция координат"
ИИ не сразу понял, что нужно плавно двигать маркеры между точками, а не "прыгать" каждые 5 секунд. Пришлось явно указать requestAnimationFrame.
- O(1) доступ к маркерам
ИИ сначала предложил массив с find() — O(n). Пришлось объяснить, что при 500 ТС и обновлениях каждые 5с это даст 100 операций в секунду. Решение: Map<id, Marker>.

Итерации

Итерация 1: ИИ использовал v-for для маркеров
❌ Проблема: при каждом обновлении Vue пересоздаёт все маркеры
✅ Решение: Убрал v-for, управляю маркерами напрямую через Leaflet API

Итерация 2: ИИ не добавил интерполяцию
❌ Проблема: маркеры "прыгают" каждые 5с
✅ Решение: Добавил requestAnimationFrame + линейную интерполяцию между точками

Итерация 3 (финальная): Использовал Map для кэша маркеров
✅ Результат: 60 FPS при 500 ТС, плавная анимация

Промпт 2: Realtime Store с минимизацией реактивных триггеров

Исходный промпт

Роль: Senior Vue 3 + Pinia Developer, эксперт по оптимизации реактивности.
Задача: Создать realtimeStore.js для управления 500+ ТС в реальном времени.

КРИТИЧНО для производительности:
1. Использовать Map<vehicleId, Vehicle> вместо Array для O(1) обновлений
2. При обновлении ТС менять ТОЛЬКО изменившиеся поля, не заменять весь объект
3. Избегать массовых пересчётов computed при каждом обновлении
4. Использовать shallowRef для больших объектов, если не нужна глубокая реактивность

Контекст обновлений:
- Каждые 5 секунд приходит 500 обновлений координат
- Каждое обновление должно триггерить ТОЛЬКО обновление соответствующего маркера
- Не должно быть перерисовки всей карты

Пример правильной логики:
```javascript
function updateVehicle(vehicleData) {
  const existing = vehicles.value.get(vehicleData.id)
  if (existing) {
    // Меняем только координаты, не весь объект
    if (existing.coordinates.lat !== vehicleData.coordinates.lat) {
      existing.coordinates.lat = vehicleData.coordinates.lat
      existing.coordinates.lon = vehicleData.coordinates.lon
    }
  }
}

Формат: ТОЛЬКО код realtimeStore.js с комментариями по оптимизациям.


### Почему был сложным

1. **Vue Reactivity Gotchas**  
   ИИ по умолчанию предлагает `vehicles.value = [...vehicles.value, newVehicle]` — это создаёт новый массив и триггерит перерисовку ВСЕХ маркеров. Пришлось объяснить разницу между мутацией и заменой.

2. **Map vs Array в Pinia**  
   Pinia + Map — неочевидная комбинация. ИИ предлагал Array + computed фильтры. Пришлось настоять на Map для O(1).

3. **Поле-by-поле обновление**  
   Ключевой инсайт: если менять `vehicle.coordinates = newCoords`, Vue триггерит обновление всех, кто смотрит на `coordinates`. Если менять `vehicle.coordinates.lat = newLat` — только тех, кто смотрит на `lat`. Это критично для производительности.

## 🎯 Промпт 3: Мок-сервер с SSE и реалистичной симуляцией

### Исходный промпт
```text
Роль: Senior Node.js Developer, эксперт по SSE и симуляции данных.
Задача: Создать mock-backend/server.js, который генерирует реалистичное движение транспорта.

Требования к симуляции:
1. 50-100 ТС, движущихся по заданным маршрутам (waypoints)
2. Обновление координат каждые 5 секунд
3. Реалистичное поведение:
   - ТС движутся от waypoint к waypoint по прямой
   - Скорость 10-40 км/ч с небольшими флуктуациями
   - 5% шанс остановки на "остановке" (статус at_stop)
   - 2% шанс задержки (delay увеличивается)
   - Загруженность салона меняется плавно (0-100%)
4. SSE endpoint /api/realtime/stream:
   - При подключении отправляет все текущие ТС
   - Каждые 5с рассылает обновления всем подключённым клиентам
   - Формат: event: vehicle_update\ndata: {...}\n\n
5. REST API для исторических данных (мок)
6. Обработка отключения клиентов (удаление из Set)

Критично:
- Использовать Set для SSE клиентов (O(1) добавление/удаление)
- Не блокировать event loop при рассылке обновлений
- Graceful shutdown при SIGTERM

Формат: ТОЛЬКО код server.js с комментариями.

Почему был сложным

- SSE protocol details
ИИ сначала забыл про event: prefix. Без него клиент не может различать типы событий. Пришлось явно указать формат.
- Реалистичная симуляция
ИИ предлагал случайные координаты в радиусе — это выглядело как "ТС летают". Пришлось объяснить концепцию waypoints и интерполяции между ними.
- Broadcasting
ИИ не учёл, что клиентов может быть несколько. Пришлось добавить Set<sseClients> и цикл рассылки.

Ключевые термины, которые пришлось использовать

| Термин | Зачем нужен |
|--------|-------------|
| Server-Sent Events (SSE) | Проще WebSocket для однонаправленного потока |
| Waypoints | Реалистичные точки маршрута, между которыми движется ТС |
| Interpolation | Плавное движение между точками, а не "прыжки" |
| Broadcasting | Рассылка обновлений всем подключённым клиентам |
| O(1) access | Критично для производительности при 500+ ТС |
| requestAnimationFrame | Плавная анимация 60 FPS |
| Reactive triggers | Минимизация перерисовок Vue |
| Map vs Array | Выбор структуры данных для O(1) обновлений |
